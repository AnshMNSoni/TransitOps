TransitOps

A Centralized, Enterprise-Grade Smart Transport Operations and Fleet Lifecycle Management Platform built on Odoo 19.

Platform Vision

TransitOps replaces manual logbooks and fragmented spreadsheets with a centralized database module designed to govern vehicle lifecycles, driver safety compliance, automated dispatch validations, real-time fleet utilization analytics, and operational profitability metrics.

It solves core scheduling conflicts, underutilized capacity, missed maintenance intervals, expired driver license liability, and lack of real-time ROI tracking in a beautiful, consolidated interface.

Visual Fleet Dashboard

The central landing page displays dynamic Bootstrap 5 responsive KPI cards reporting active metrics:

  • Fleet Utilization (%): Active vehicles on trip / total fleet capacity.
  • Vehicle Summary: Live counts of Available, On Trip, and In Shop (Maintenance) vehicles.
  • Operations Summary: Counts of active trips, pending dispatches, and drivers currently on duty.

Core Model Architecture

Model Class Purpose & Primary Fields Enforced Business Rules
transit.vehicle Registry for vehicle status, odometer, acquisition cost, and dimensions. Unique registration number, automatic cost and ROI updates, status pool management.
transit.driver Driver safety score, license number, license category, and expiry date. Unique license number, automatic age/expiry verification, dispatch blocking rules.
transit.trip Dispatches cargo trips from source to destination. Links driver and vehicle. Capacity constraints checks, double-booking prevention, automated state sync.
transit.maintenance Tracks repair tasks, service costs, and maintenance durations. Hides vehicle from dispatch pool while in active status, updates operational costs on complete.
transit.fuel.log Logs fuel capacity, cost per liter, and linked trips. Automatically updates total vehicle operational costs and overall fuel efficiency.

End-to-End Transport Operations Lifecycle

Follow the steps below inside the browser interface to test the complete lifecycle workflow of vehicles, drivers, and dispatches:

1. Registry Input
Go to Fleet and register a vehicle (e.g. `Van-05`, max capacity `500` kg). Go to Drivers and register a driver (e.g. `Alex`, with valid future license expiry). Set both to Available.
2. Smart Dispatch Validation
Create a new record in Trips. Assign your vehicle and driver, and set Cargo Weight to `600` kg. Click Dispatch. The system will throw a capacity error. Change weight to `450` kg and dispatch. The trip shifts status to Dispatched.
3. Automated Status Lock
Once dispatched, open your vehicle and driver profiles. The system has automatically shifted their status to On Trip, removing them from any other trip dispatch selection pool.
4. Mandatory Completion Wizard
Open the dispatched trip and click Complete. This triggers the mandatory completion popup, forcing the user to log: Final Odometer, Actual Distance, Fuel Consumed, and Fuel Price.
5. Post-Trip Log Automation
Upon wizard confirmation: the trip moves to completed, the vehicle's odometer is updated, and both vehicle and driver return to Available. A fuel log is automatically registered for the trip's cost.
6. Active Maintenance Lockout
Go to Maintenance and log a repair for the vehicle. Shifts vehicle status to In Shop, removing it from dispatch selector pool. Complete maintenance to return it to Available.
7. Analytics, ROI, and Reports
Check the Dashboard for utilization cards. Go to Analytics to see vehicle bar charts and Vehicle ROI (%). Click Print on a vehicle to export the PDF Fleet Summary report.

Running the Stack

To spin up Odoo 19 and Postgres 17 database containers, ensure your directory contains the docker-compose.yml file and execute:

docker compose up -d

To shut down the application stack and volumes:

docker compose down -v

Running Automated Tests

We wrote a comprehensive integration test suite verifying registrations, dispatch validations, wizard inputs, and dashboard aggregates. Run it inside the container using:

docker exec -u 0 odoo19_app odoo -d odoo19 -u transit_ops --test-tags=transit_ops --stop-after-init --http-port=8070

A clean exit code with 0 failed, 0 errors confirms compliance.

Docker Configuration Schema

The system leverages environment variables to run isolated container instances. Set up your local .env file mirroring the template below:

# Postgres DB Credentials
POSTGRES_USER=odoo
POSTGRES_PASSWORD=odoo_db_password_sec
POSTGRES_DB=postgres

# Odoo Server Configuration
HOST=db
USER=odoo
PASSWORD=odoo_db_password_sec
PORT=5432
HTTP_PORT=8069
